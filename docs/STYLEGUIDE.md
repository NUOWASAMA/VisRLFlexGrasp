# 项目开发全局规约
版本：V1.0 | 适用：本项目全体开发成员
## 第 1 章 总则
### 1.1 规约目的与作用
* 统一项目全流程的开发标准与协作约定，降低团队沟通成本，避免代码风格混乱、目录结构失控、环境配置不一致等问题，保障项目可维护性与可迭代性，为项目结题、代码交付与后续扩展提供统一基准。

### 1.2 适用范围与覆盖场景
* 本规约覆盖本项目所有开发交付物，包括但不限于：
  * 项目源代码、配置文件、脚本工具
  * Git 仓库分支、提交记录、合并流程
  * 项目文档、架构图、接口说明
  * 仿真环境与实体设备的开发调试流程
  * 所有项目成员在提交代码、更新文档、协作开发时均需遵守本规约。

### 1.3 执行优先级与异议处理
* 本规约为项目强制开发标准；规约有明确要求时优先遵循；未覆盖的编码细节，遵循对应语言官方规范（Python 遵循 PEP8、C++ 通用工业编码规范）。特殊场景无法满足规约要求，需在 PR 评审中书面说明理由。
* 对规约条款有异议时，由项目负责人组织团队讨论，达成一致后统一修订，禁止个人私自突破规约。

### 1.4 全局术语与缩写统一字典
* 本项目所有代码、文档、接口字段必须使用统一术语，禁止同物异名。核心术语在此处统一登记，新增术语需同步更新：

| 术语 / 缩写 | 全称 | 统一含义             |
| :---------- | :--- | :------------------- |
| Gripper     | -    | 机械末端夹爪执行器   |
| Grasp Pose  | -    | 机器人抓取位姿       |
| Detection   | -    | 视觉目标检测算法     |
| Sim         | -    | CoppeliaSim 仿真环境 |
| World Frame | -    | 世界基准坐标系       |
| RL          | -    | 强化学习             |

## 第 2 章 工程文件与目录规范
### 2.1 文件与文件夹命名规则
* 全局统一使用 小写字母 + 下划线（snake_case） 命名，禁止使用中文、空格、特殊符号、大小写混排。
* 文件名需清晰表达文件用途，禁止使用 1.py、test.cpp、新建文件夹 等无意义命名。
* 临时调试文件命名以 tmp_ 前缀开头，严禁提交至远程仓库。

### 2.2 标准目录架构与存放约束
* 项目根目录必须遵循以下固定结构，新增顶层目录需经团队讨论确认：

```
VisRLFlexGrasp/
├── frontend/        # 前端交互层：UI界面、可视化、交互逻辑
├── backend/         # 后端调度层：接口服务、业务调度、中间件
├── algorithm/       # 算法核心层：视觉识别 + 强化学习决策
├── embedded/        # 嵌入式C++控制层：机械臂控制、通信、驱动
├── database/        # 数据层：数据表结构、迁移脚本、初始化数据
├── common/          # 跨模块公共组件：工具类、协议定义、通用逻辑
├── docs/            # 项目文档：开发规范、部署指南、接口说明、架构图
├── deploy/          # 部署脚本：一键启动、环境配置、服务编排
├── tests/           # 测试用例：单元测试、集成测试、全链路验证
└── README.md        # 项目总览与快速启动说明
```
* 代码文件严格按业务分层归属至对应顶层模块目录：frontend/、backend/、algorithm/、embedded/、common/，禁止在项目根目录随意散落业务代码、脚本文件。
* 所有项目文档、开发规约、部署说明、架构图表、接口文档统一归类存放于 docs/ 目录并按类别分子目录管理，严格执行代码与文档物理隔离原则，禁止文档与业务代码混放。

### 2.3 路径编写统一约定
* 代码、文档、脚本中禁止使用绝对路径，所有资源引用必须使用相对路径。
* 路径分隔符统一使用正斜杠 /，禁止 Windows 反斜杠 \，保障跨平台兼容。

### 2.4 资源、配置与第三方库存放规则
* 配置、静态资源、第三方依赖均分散在各业务模块内部管理，规范如下：
  * 配置文件管理
    * 各模块内置独立 config/ 子目录，按 base/dev/sim/real 划分多环境配置；
    * 禁止在代码中硬编码参数、设备地址、密钥等内容。  
    * 敏感私有配置使用 .env 文件本地存放，仅提交 .env.example 模板至仓库，真实配置不入库。
  * 静态资源管理
    * 静态资源随所属模块存放，不统一归集到项目根目录：
        * 前端图片、图标：frontend/src/assets/
        * 视觉模型、数据集：algorithm/assets/
        * 仿真场景、机械臂模型文件：embedded/assets/
    * 单文件体积超过 10MB 的模型、数据集禁止提交 Git 仓库，采用外部存储链路分发。
  * 第三方依赖管理
    * 优先使用包管理工具在线拉取依赖（Python 使用 pip、C++ 使用 CMake FetchContent）；
    * 无法通过包管理器安装的本地库，存放至对应模块下的 third_party/ 子目录，并配套记录依赖版本、编译说明文档。

### 2.5 临时文件与构建产物管理
* 编译产物、缓存文件、临时日志、调试生成的中间文件、各模块编译生成的 build 目录，必须全部加入 .gitignore，禁止提交至远程仓库。

* 废弃代码、过期文件直接删除，禁止保留 xxx_old.cpp、xxx_bak.py 等备份文件，历史版本通过 Git 回溯。
* 本地临时调试文件统一以 tmp_ 开头命名，确保全局 .gitignore 可自动过滤。

## 第 3 章 源代码编写规范
### 3.1 标识符命名统一标准
* 所有变量、函数、类、接口字段命名，必须严格遵循 1.4 全局术语字典，禁止对同一业务概念自定义别名。
* 项目内全语言遵循统一命名规则，保障跨模块可读性。

#### 3.1.1 变量与成员属性
* 普通变量、函数参数：统一使用 snake_case（小写 + 下划线）
* C++类私有成员属性：加 m_ 前缀，如 m_joint_angle
* Python 类私有/保护成员：遵循 PEP8，使用单下划线 _ 前缀，如 _joint_angle
* 禁止使用单字符变量名（循环计数器 i/j/k 除外）

#### 3.1.2 函数与类方法
* 统一使用 snake_case（小写 + 下划线）
* 命名采用[动词 + 名词]结构，明确表达函数功能，如 calculate_grasp_pose ()
* 布尔返回值函数以 is_/has_/can_ 开头，如 is_collision_detected ()

#### 3.1.3 类、结构体、枚举
* 类与结构体：
   * 统一采用 PascalCase（大驼峰命名），每个单词首字母大写。
     示例：`GraspPlanner`、`RobotState`、`TcpClient`。
   * C++ 中的结构体与类遵循相同命名规则，不额外增加前缀或后缀。
* 枚举类型：
   * C++ 枚举类型名称：采用 `E` 前缀 + PascalCase，例如 `EGripperStatus`、`EControlMode`。此规则同时适用于 `enum` 和 `enum class`。
   * Python 枚举类名称：直接采用 PascalCase，不添加额外前缀，例如 `GripperStatus`、`ControlMode`（保持 Python 原生惯用法）。
   * 所有枚举值：统一采用 UPPER_SNAKE_CASE（全大写下划线），例如 `OPEN`、`CLOSED`、`MAX_RETRY_COUNT`。
     注意：枚举值命名必须与 3.1.4 节定义的全局常量命名风格保持一致，避免在同一作用域内出现命名冲突。
* 名空间与模块：
   * C++ 命名空间统一使用 snake_case（小写下划线），与目录层级对应，例如 `namespace vis_rl_grasp::algorithm`。
   * Python 包/模块名同样使用 snake_case，遵循 PEP8 规范。

#### 3.1.4 常量与宏定义
* 统一使用 UPPER_SNAKE_CASE（全大写 + 下划线），如 MAX_SPEED_LIMIT
* 禁止在代码中直接使用魔法数字（无字面定义的固定数值），所有固定数值必须定义为常量并加注释说明含义

### 3.2 代码格式与语法约束
* 缩进统一使用 4 个空格，禁止使用 Tab 字符；行宽上限 120 字符，超出需合理换行。
* 函数、类、控制语句的左大括号独占一行，右大括号与开头对齐；禁止单行写复杂逻辑。
* 运算符前后、逗号后必须加空格，保障代码可读性。
* 推荐单个函数长度不超过 80 行，单个文件不超过 500 行，超出需拆分模块。

### 3.3 模块导入与依赖引用规则
* 导入顺序按[标准库 → 第三方库 → 项目内部模块]排列，组之间空一行分隔。
* 禁止循环依赖，禁止导入未使用的模块，禁止使用 import * 通配符导入。
* 模块依赖遵循单向原则，底层工具模块不得反向依赖上层业务模块。

### 3.4 分层解耦与调用边界
* 项目代码分为四层，严格遵守调用方向，禁止跨层耦合：
  * 工具层：通用工具函数、数学库、日志模块，对应项目 common/
  * 适配层：仿真接口、硬件驱动、通信封装，对应 embedded/comm 硬件驱动模块
  * 业务层：核心算法、业务逻辑、状态机，对应 algorithm/
  * 应用层：主程序、任务调度、交互入口，对应 backend/、frontend/

* 上层可调用下层，禁止下层反向调用上层；业务层禁止直接操作硬件 / 仿真接口，必须通过适配层调用。

### 3.5 注释编写统一标准
* 文件头注释：每个源文件顶部必须标注文件功能、作者、创建日期。
* 函数注释：所有对外暴露的函数必须加注释，说明功能、入参含义、返回值、异常场景。
* 行内注释：仅对复杂逻辑、特殊处理、边界条件加注释，禁止复述代码的无效注释。
* 注释统一使用中文编写，专业术语可保留英文，禁止中英文口语化混排。

### 3.6 异常处理与容错要求
* 所有对外接口、核心函数必须做入参合法性校验，异常输入需返回明确错误，禁止程序直接崩溃。
* 指针、对象使用前必须判空，数组、容器访问前必须校验下标 / 长度，禁止越界访问。
* 捕获异常后必须记录日志并处理错误，禁止静默吞掉异常；调试阶段禁止用 try-catch 掩盖代码问题。

### 3.7 日志分级输出规范
* 日志统一分为四级，严格按级别输出：
* DEBUG：调试细节信息，仅开发环境开启，正式版本关闭
* INFO：正常流程关键节点，如模块初始化、任务启动完成
* WARN：异常但不影响运行的警告，如参数超限自动修正、一次通信重试
* ERROR：功能失效、流程中断的严重错误，必须附带错误原因与位置
* 禁止在循环中高频打印日志，禁止用打印替代日志输出。

## 第 4 章 模块通信接口规约
### 4.1 接口设计基本原则
* 单一职责：一个接口只完成一项功能，禁止一个接口兼容多种无关业务。
* 向后兼容：接口修改时优先新增字段，禁止直接删除、重命名已有字段，避免影响其他模块。
* 输入必校验：所有接收方必须校验接口字段合法性，禁止默认信任上游数据。

### 4.2 数据报文统一结构
* 模块间通信（TCP/JSON）统一使用标准报文结构，禁止自定义零散格式：

```json
{
  "code": 0,
  "msg": "success",
  "data": {},
  "timestamp": 1700000000000
}
```
code：错误码，0 代表成功，非 0 代表异常
​msg：错误描述或状态说明
​data：业务数据体
​timestamp：发送端时间戳，毫秒级统一约定

### 4.3 字段命名与数据类型约定
* 所有报文字段统一使用 snake_case 命名，与代码变量命名保持一致。
* 数据类型必须明确：数值区分 int/float，布尔值使用 true/false，禁止用 0/1 替代布尔。
* 可选字段必须明确标注默认值，禁止字段缺失导致接收方解析崩溃。

### 4.4 统一错误码定义规则
* 错误码为整数，按模块分段，全局统一维护：
* 0：成功
* 1000-1999：通用错误（参数错误、格式错误、超时）
* 2000-2999：算法模块错误
* 3000-3999：硬件 / 仿真模块错误
* 4000-4999：通信模块错误
* 新增错误码必须同步更新全局错误码字典，禁止随意定义。

### 4.5 接口变更与弃用流程
* 新增接口：需提前同步给所有对接模块，附带接口文档，联调通过后正式启用。
* 改接口：优先兼容旧版本，至少保留 1 个版本的过渡期，禁止直接不兼容修改。
* 用接口：提前标记 deprecated，通知所有使用方迁移，确认无人使用后再删除。

### 4.6 超时、重试、重连机制规范
* 通信默认超时时间 3 秒，超时后按业务场景决定是否重试。
* 机制：单次请求最多重试 3 次，每次间隔 1 秒，连续失败后上报错误。
* 接断开后自动重连，重连间隔逐步递增（1s/3s/5s），重连过程中缓存关键数据，禁止程序退出。

## 第 5 章 Git 版本管理规范
### 5.1 分支体系与使用规则
* 支模型，仅保留四类分支，禁止随意创建无名分支：
* in：主分支，存放稳定可运行版本，仅允许从 dev 合并，禁止直接提交。
* v：开发主分支，汇总所有功能开发，保持可编译运行状态。
* ature/xxx：功能开发分支，从 dev 切出；
* gfix/xxx：缺陷修复分支，从 dev 切出；
* 分支统一采用 `前缀/功能描述` 格式，前缀固定 feature /bugfix，斜杠后描述使用小写 + 下划线，禁止使用 dev2、test 等无意义分支名。

### 5.2 Commit 提交信息统一格式
提交信息必须遵循以下格式，禁止随意写无意义描述：
```
<类型>: <简明描述>
```
* 类型固定为以下选项：
  * feat：新增功能 / 模块
  * fix：修复 bug
  * docs：文档更新
  * style：代码格式调整（不影响逻辑）
  * refactor：代码重构（无功能变更）
  * test：测试代码相关
  * chore：构建脚本、依赖配置等工程事务
  * 例：feat: 完成机械臂逆运动学求解模块、fix: 修复 TCP 通信丢包导致的解析崩溃
  单次提交只做一件事，禁止把多个无关功能打包在一次提交中。

### 5.3 PR 合并与代码评审要求
* 所有功能分支合并入 dev，必须提交 Pull Request，禁止直接 push 到 dev/main。
* PR 至少经过 1 名其他成员评审通过，项目负责人确认后方可合并。
* 评审发现的问题必须修改完成后再次提交，禁止带问题合并代码。

### 5.4 禁止提交内容黑名单
* 以下内容绝对禁止提交至远程仓库：

1. 编译产物、缓存文件、临时文件、本地 build 目录、本地日志
2. IDE 配置文件、个人环境配置文件、本地.env 敏感配置文件
3. 包含密码、密钥、设备私密参数的配置文件
4. 超过 10MB 的大文件、数据集、模型权重
5. 与项目无关的个人文件、临时测试代码

### 5.5 版本标签与发布标记规则
* 项目里程碑节点（如中期检查、联调完成、结题版本）必须打 Tag 标记：
  * 标签格式：vX.Y.Z，如 v1.0.0
  * X：主版本号，架构大变更时递增
  * Y：次版本号，新增核心功能时递增
  * Z：修订号，bug 修复、小优化时递增
* 打 Tag 时附带版本说明，明确该版本核心功能与已知问题。

## 第 6 章 依赖、环境与部署规范
### 6.1 项目依赖版本锁定规则
* 所有第三方依赖必须锁定精确版本，禁止写模糊版本号。
* ython：使用 requirements.txt 固定版本，如 numpy==1.24.3
* C++Make 中明确指定依赖最低版本，第三方库统一版本号
* 依赖版本必须经团队确认，验证无兼容性问题后更新版本锁定文件，禁止个人私自升级。

### 6.2 配置文件分级管理规则
* 文件按环境分级管理，禁止一套配置混用所有环境：
  * base：基础公共配置，所有环境通用
  * dev：本地开发环境配置
  * sim：仿真环境配置
  * real：实体设备环境配置
* 敏感配置（端口、密钥、设备参数）使用 .env 文件存放，提供 .env.example 模板，真实配置不入库。

### 6.3 多环境隔离约定（开发 / 仿真 / 实体）
* 三个环境完全隔离，通过环境变量或启动参数切换，禁止在代码中硬编码环境开关。
* 开发环境仅用于本地调试，仿真环境用于算法验证，实体环境仅在硬件联调时使用。
* 禁止在实体设备环境中运行调试代码、打印高频日志。

### 6.4 仿真与实体设备兼容要求
* 业务层代码必须同时兼容仿真与实体环境，通过适配层屏蔽底层差异，上层业务无感知。
* 新增功能必须先在仿真环境验证通过，再部署到实体设备，禁止直接在硬件上调试核心逻辑。
* 仿真与实体的接口参数、数据格式必须完全一致，禁止两套接口定义。

## 第 7 章 项目文档撰写规范
### 7.1 Markdown 统一排版标准
* 标题分级清晰，一级 # 仅用于文档标题，正文按 ### / #### / ##### 层级递减，禁止跳级使用标题。
* 代码块必须标注语言类型，关键内容使用加粗突出，禁止大段无格式文本。
* 列表、表格格式对齐，中英文之间保留 1 个空格，标点符号统一使用中文标点（代码 / 术语除外）。

### 7.2 文档命名与存放规则
* 文档统一使用 snake_case 命名，后缀为 .md，如 tcp_communication_spec.md。
* 文档按类别放入 docs/ 对应子目录，如 docs/architecture/、docs/api/、docs/deploy/。
* 文档名称明确表达文档内容，禁止使用 说明文档.md、新版文档.md 等模糊命名。

### 7.3 架构图与流程图绘制规范
* 优先使用 Mermaid 语法绘制架构图、流程图、时序图，内嵌到 Markdown 文档中，便于版本管理。
* 复杂图表可使用 Draw.io 绘制，源文件与导出图片一同放入 docs/assets/ 目录。
* 图表必须逻辑清晰、标注完整，禁止无图例、无说明的模糊示意图。

### 7.4 文档同步更新要求
* 代码变更涉及接口、架构、配置时，必须同步更新对应文档，代码评审同时检查文档更新。
* 禁止代码已上线、文档仍为旧版本；过期内容及时标记或删除，避免误导后续开发。
